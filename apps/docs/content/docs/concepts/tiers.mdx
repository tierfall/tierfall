---
title: The four tiers
description: TierFall recognizes four routing tiers. Each captures a cost / capability / availability tradeoff.
---

TierFall classifies every adapter into one of four tiers. Index `0` is the most expensive; index `3` is the cheapest. The router _falls_ from low-index to high-index — never the other way without explicit opt-in.

```ts
export const TIERS = ['premium-cloud', 'cheap-cloud', 'self-hosted-edge', 'on-device'] as const;
```

## The tiers

| Tier               | Index | Typical providers                             | Cost shape             | Notes                                      |
| ------------------ | ----- | --------------------------------------------- | ---------------------- | ------------------------------------------ |
| `premium-cloud`    | 0     | Claude, GPT-5                                 | Highest                | First attempted by default                 |
| `cheap-cloud`      | 1     | DeepSeek, Groq, Cerebras                      | Cheap per-token        | Fan-out via the OpenAI-compatible adapter  |
| `self-hosted-edge` | 2     | vLLM, LM Studio, self-hosted Anthropic compat | Free + your infra cost | OpenAI-compatible against your private URL |
| `on-device`        | 3     | Ollama, llama.cpp                             | Free                   | Lives on the caller's machine              |

A single adapter package can serve multiple tiers depending on configuration. `@tierfall/adapter-openai-compatible` is the clearest example — point it at OpenAI for premium, at Groq for cheap, at vLLM for self-hosted-edge. The `tier` field on the adapter instance is what the policy actually sees.

## Why this order?

Falling toward cheaper composes naturally with budget caps: if a premium adapter throws `BudgetExceededError`, falling to a cheaper tier is the right move — the next adapter is cheaper by definition. Climbing the other direction risks runaway cost in the failure path.

The tier hierarchy also encodes a rough capability-vs-availability tradeoff. Frontier models live in premium; smaller fast models live in cheap; local quantized models live on-device. The router's job is to find the highest-fidelity tier that's actually available _and_ within budget _and_ capable of the request — falling progressively until one fits.

## Each adapter declares its tier

```ts
class AnthropicAdapter implements Adapter {
  readonly tier: Tier = 'premium-cloud';
  // ...
}

class OllamaAdapter implements Adapter {
  readonly tier: Tier = 'on-device';
  // ...
}

class OpenAICompatibleAdapter implements Adapter {
  readonly tier: Tier; // per-instance from config.tier ?? 'cheap-cloud'
  // ...
}
```

Per-instance tier matters for `@tierfall/adapter-openai-compatible` because the same code talks to providers at different tier levels. The [presets sub-export](/docs/reference/adapter-openai-compatible#presets) hard-codes the right tier per provider (OpenAI = premium, DeepSeek = cheap, etc.).

## Capability shape

Every adapter declares an `AdapterCapability` block. The policy reads it to filter unsuitable adapters out before the router even gets the request.

```ts
interface AdapterCapability {
  readonly contextWindowTokens: number;
  readonly supportsTools: boolean;
  readonly supportsStreaming: boolean;
  readonly supportsStructuredOutput: boolean;
  readonly costPerMillionInputTokens: number | null;
  readonly costPerMillionOutputTokens: number | null;
}
```

In v0.1, `supportsTools` / `supportsStreaming` / `supportsStructuredOutput` are conservatively `false` on every adapter. The capability is a contract about what _the adapter_ does today, not what the underlying model could theoretically do. v0.4 will flip these flags alongside wire-level implementation.

`null` costs (typically `on-device`) indicate free tier and are treated as `0` in the policy's cost estimator.
