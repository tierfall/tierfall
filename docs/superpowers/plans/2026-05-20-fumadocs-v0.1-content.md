## Filter vs fall — they look similar but they're different

[SECTION — the demo's Scenario 2 narration explains this well.
Cross-reference policy filtering (silent) vs router fall (recorded).]

|                       | Filter                       | Fall                     |
| --------------------- | ---------------------------- | ------------------------ |
| Who does it           | `DefaultPolicy.evaluate`     | `Router.complete`        |
| When                  | Before any HTTP              | After adapter throws     |
| Recorded in fallChain | No                           | Yes (one entry per fall) |
| Why                   | Capability/budget pre-flight | Runtime failure          |

## How the router falls

\`\`\`mermaid
sequenceDiagram
participant U as Caller
participant R as Router
participant P as premium-cloud
participant C as cheap-cloud
participant L as on-device

    U->>R: complete(request)
    R->>P: try complete
    P-->>R: ProviderUnavailableError
    Note right of R: record FallDiagnostic<br/>reason='provider-unavailable'
    R->>C: try complete
    C-->>R: BudgetExceededError
    Note right of R: record FallDiagnostic<br/>reason='budget'

    L-->>R: { text, usage, ... }

## What the policy does

[INTRO — policy is a pure function: (request, adapters) → ordered Adapter[].
Filters by capability requirements; sorts by tier-index ascending.]

\`\`\`ts
import { DefaultPolicy, Router } from '@tierfall/core';

const policy = new DefaultPolicy();
const ordered = policy.evaluate(request, adapters);
const router = new Router(ordered);
const response = await router.complete(request);
\`\`\`

## Filters

[TABLE of 5 filter triggers from DefaultPolicy spec §4.1.]

| Request field                        | Excludes adapter when                             |
| ------------------------------------ | ------------------------------------------------- |
| `requires.minContextWindowTokens: N` | `capability.contextWindowTokens < N` (strict `<`) |
| `requires.tools: true`               | `capability.supportsTools === false`              |
| `requires.streaming: true`           | `capability.supportsStreaming === false`          |
| `requires.structuredOutput: true`    | `capability.supportsStructuredOutput === false`   |
| `maxCostUSD: C`                      | `estimateCost(capability) > C` (strict `>`)       |

[ONE PARAGRAPH explaining that filters combine with AND — adapter must pass every active filter to survive.]

## Cost estimation

[SECTION explaining the 500 input + 500 output token budget for v0.1.
Show the formula.]

function estimateCost(cap: AdapterCapability): number {

const outputCost = ((cap.costPerMillionOutputTokens ?? 0) / 1*000_000) * 500;
return inputCost + outputCost;
}

## Contract

[INTRO — exported from @tierfall/core. Every adapter implements this
interface. v0.1 ships three adapter packages; you can author your own.]

\`\`\`ts
import type { Adapter, AdapterCapability, Tier } from '@tierfall/core';

interface Adapter {
readonly name: string;
readonly tier: Tier;
readonly capability: AdapterCapability;
complete(request: LLMRequest): Promise<LLMResponse>;
}
\`\`\`

## Field meanings

[SHORT TABLE describing name, tier, capability, complete.]

| Field        | Purpose                                                      |
| ------------ | ------------------------------------------------------------ |
| `name`       | Identifies adapter instances in `FallDiagnostic.adapterName` |
| `tier`       | Declares which routing tier this adapter serves              |
| `capability` | What this adapter can/can't do; consumed by `DefaultPolicy`  |
| `complete`   | Executes the request; throws typed errors on failure         |

## What an adapter MUST throw

[TABLE of error contracts.]

| Condition                                                         | Error                                                  |
| ----------------------------------------------------------------- | ------------------------------------------------------ |
| Network failure / 4xx / 5xx (not rate limit) / malformed response | `ProviderUnavailableError`                             |
| Rate limit / quota exceeded                                       | `BudgetExceededError`                                  |
| Request needs capability the adapter doesn't have                 | `CapabilityMismatchError`                              |
| Anything else                                                     | Any `Error` (Router classifies as `reason: 'unknown'`) |

## What the Router does with the response

[ONE PARAGRAPH — Router overrides `tier` and `fallChain` on the response;
adapter's own values for those fields are ignored. Adapter is responsible
for `text`, `model`, `usage`.]

## Authoring a new adapter

[BRIEF section — implement the interface, add to your project's adapter list,
